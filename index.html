Part 1: Fundamentals

Beyond Autocomplete: The Agentic Loop

Unlike standard AI coding tools that wait for you to highlight code, Claude Code thinks before it acts. It operates autonomously through a three-phase continuous loop.

1

Explore

The agent navigates your directory using native Bash commands. It reads relevant files, parses your package.json, and builds a contextual map of your architecture.

2

Plan

Using features like Extended Thinking (Ultrathink), Claude pauses to draft a strategy before modifying code. This critical step drastically reduces AI-generated technical debt.

3

Execute

It edits files, runs terminal commands (like compiling), and verifies its own work by running your test suite. If tests fail, it loops back to step 1 to debug.

Ecosystem: Agents, Skills & Collaboration

Claude Code is not a single monolith. It is an ecosystem where a Lead Agent orchestrates specialized Subagents and dynamically loads Skills to solve complex engineering problems.

The Agent (Orchestrator)

The Lead Agent is the brain. It maintains the overarching context of your prompt, decides which files to read, and delegates bounded tasks to specialized Subagents (like an exploration agent or a testing agent).

The Skill (Domain Knowledge)

A Skill is a prompt-based meta-tool. Instead of hardcoded plugins, a Skill is just a directory with markdown files (e.g., SKILL.md). By using Progressive Disclosure, Claude dynamically alters its own system prompt and tool permissions based on the skill loaded.

The Collaboration

The true power is in orchestration. The Lead Agent can load the golang-pro Skill to understand Go idioms, utilize an MCP Server to securely fetch database schemas, and then spin up a Subagent to write concurrent tests.

Context is Everything: The CLAUDE.md File

The biggest pitfall of AI coding is accumulating technical debt. Prevent this by treating the AI like a new hire: provide an onboarding document.

Placing a CLAUDE.md file in your project root sets persistent boundaries. Claude reads this automatically on startup to understand your architectural rules, formatting preferences, and test mandates.

  • Enforce Clean Architecture or MVC patterns.
  • Mandate that tests must be run after every edit.
  • Protect specific directories (e.g., "Never modify DB migrations").
CLAUDE.md (Starter Template)
# Project Onboarding

## 1. Architecture
- We use strict Clean Architecture.
- UI components go in `/web/components`.
- Business logic goes in `/internal/core`.

## 2. Testing Mandates
- ALWAYS run `npm run test:unit` after edits.
- Ensure test coverage does not drop.

## 3. Strict Rules
- NEVER modify files in `/migrations` directly.
- Avoid deeply nested conditionals.
- Use early returns.

Initialize Your First Workspace From Scratch

Follow these precise steps to properly configure your environment. Setting up correctly prevents common PATH and execution errors.

1

System Prerequisites

Ensure Node.js (npm) is installed.

Windows Users: Claude Code requires WSL2 (Windows Subsystem for Linux). Do not run it in standard Command Prompt. For best performance, keep your project files in the WSL filesystem (~/), not mounted Windows drives (/mnt/c/).

2

Install the CLI

Install Claude Code globally via npm to make it accessible anywhere in your terminal.

npm install -g @anthropic-ai/claude-code
3

Authenticate

Link your terminal to your Anthropic Console. Running this command opens a secure browser window for OAuth validation.

claude login
4

Initialize Session

Navigate to your project root. Typing claude drops you into the interactive agent interface.

cd ~/my-project
claude

Extend Claude: Build & Setup MCPs and Skills

Give your agent access to the real world. Learn how to configure official MCP servers, build custom Skills from scratch, and orchestrate them dynamically.

1. Setup MCP Servers

Model Context Protocol (MCP) servers connect Claude to external APIs, databases, and tools. Official MCPs like Firecrawl (for web scraping) or GitHub can be added instantly.

How to add newly: Create a .mcp.json file in your project root, or use the CLI command claude mcp add.

// .claude/settings.json
{
  "mcpServers": {
    "firecrawl": {
      "command": "npx",
      "args": ["-y", "@firecrawl/mcp-server"],
      "env": { "FIRECRAWL_API_KEY": "YOUR_KEY" }
    }
  }
}

2. Build & Add Skills

A Skill is a prompt-based meta-tool that teaches Claude specific workflows or standards (e.g., golang-pro). They act as progressive disclosure, injecting context only when needed.

How to build & add: Create a directory inside .claude/skills/. Inside, create a SKILL.md file with YAML frontmatter. Invoke it via /skill load my-skill.

# .claude/skills/api-reviewer/SKILL.md
---
name: api-reviewer
description: "Use this skill to audit REST endpoints."
---
# Instructions
When reviewing API endpoints:
1. Ensure all routes use plural nouns.
2. Check that the DB optimizer MCP was used.
3. Validate OpenAPI schema in `/docs`.

Advanced: Hooks, Permissions, Slash Commands & Worktrees

Once you're comfortable with the basics, these features unlock the real productivity gains. This is where Claude Code stops being a chatbot and starts being a system integrator.

Hooks

Shell commands the harness runs on lifecycle events (PreToolUse, PostToolUse, Stop, SessionStart). Perfect for auto-formatting, lint gates, or notifications — because hooks run in your shell, not Claude.

// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "npm run lint --silent"
      }]
    }]
  }
}

Permission Modes

Four modes control how much autonomy Claude gets. Start with default, use plan for exploratory work, and only opt into bypass inside sandboxes or throwaway containers.

  • default — ask before every risky tool call (safest)
  • acceptEdits — auto-approve Edit/Write within allowlist
  • plan — read-only, no mutations anywhere
  • bypass — full autonomy (sandbox only!)

Custom Slash Commands

Drop a markdown file into .claude/commands/ and it becomes a reusable slash command. Arguments via $ARGUMENTS. Great for repetitive flows: code review, release notes, bug triage.

# .claude/commands/review.md
---
description: Review staged changes
---
Run `git diff --staged`, then critique the
changes for: correctness, naming, missing
tests, and edge cases. Focus on $ARGUMENTS.

Invoke: /review auth layer

Subagents

A subagent is a specialized agent with its own tools, system prompt, and (optionally) its own git worktree. Use them to parallelize independent work or to keep focused searches out of your main context.

# .claude/agents/Explore.md
---
name: Explore
description: Fast codebase search
tools: [Read, Grep, Glob, Bash]
---
You are a read-only researcher. Return
concise findings with file:line refs.
Never edit files.

Plan Mode vs Extended Thinking

These are frequently confused. They solve different problems:

  • Plan mode — Claude investigates read-only, then surfaces a written plan for you to approve before any edits.
  • Extended Thinking — the model spends extra tokens reasoning internally before replying. No user checkpoint.

Rule of thumb: Plan mode for risky refactors you want to review. Extended thinking for hard problems the model keeps getting wrong.

Git Worktrees

Run agents in parallel on separate branches without touching your working tree. Each agent gets an isolated checkout — no context collision, no merge chaos.

# Agent A: experiment on a feature
git worktree add ../feat-auth feat/auth

# Agent B: refactor in parallel
git worktree add ../refactor-db refactor/db

Orchestrating Polyglot Microservices

Working in a complex repository? Learn how to configure Claude Code as a Lead Agent to dispatch tasks across iOS, Android, Go, and .NET pipelines.

1. The Root Dispatcher (Fundamental)

Create a master CLAUDE.md at the root of your project. This acts as the "router." Instruct Claude to identify which microservice needs changing and to load the appropriate domain-specific skill before writing code.

Root /CLAUDE.md (Orchestrator Pattern)
# Monorepo Orchestration Rules

// When working across the stack, process sequentially:

## 1. .NET Backend (/backend-api)
- ALWAYS load the .NET Clean Architecture skill first.
- Rebuild via `dotnet build` before moving to other layers.

## 2. Go Microservices (/workers)
- Load the `golang-pro` skill.
- Ensure all gRPC interfaces match the updated .NET API.

## 3. Mobile Clients (/ios & /android)
- For iOS: Use `XcodeBuildMCP` to verify Swift changes.
- For Android: Ensure UI uses Jetpack Compose patterns.
- NEVER mix Swift syntax into Kotlin files or vice-versa.

2. Platform-Specific Tooling

Equip the agent with the right tools for each environment using MCP and local skills:

📱
iOS/Swift
Integrate XcodeBuildMCP for building & simulating.
🤖
Android/Kotlin
Load robust skills for MVVM + Jetpack Compose.
☁️
Go Services
Load golang-pro for strict concurrency standards.
⚙️
.NET Core
Enforce strict Clean Architecture rules.

Prompt Claude sequentially to act as a full-stack orchestrator:

Pro Tips for Beginners

Avoid common pitfalls and technical debt. Here are the golden rules to ensure a smooth transition into agentic development.

1. Commit Before You Prompt

Because Claude Code modifies files directly, always commit your current git state before giving it a complex task. If the agent misunderstands and breaks something, a quick git reset --hard is your ultimate safety net.

2. Manage Context Aggressively

AI gets confused in long, sprawling conversations. Use the /clear or /compact commands frequently to reset the agent's memory. This keeps costs down and ensures the agent stays laser-focused on the immediate task.

3. Step-by-Step Orchestration

Don't ask "Build me a login system." Instead, orchestrate the process:
1. "Explore the auth folder."
2. "Draft a plan for a JWT login system."
3. "Execute step 1 of the plan."

4. Leverage the /help Command

Claude has its own documentation built-in. If you don't know how to do something, just type /help or ask the agent directly: "How do I add an MCP server for PostgreSQL?"

Frequently Asked Questions

Common questions and troubleshooting tips for getting started with Claude Code.

Why does claude command not found after install?

Your npm global bin directory may not be in your PATH. Run npm config get prefix and add the resulting bin/ directory to your shell profile (~/.zshrc or ~/.bashrc).

Claude keeps asking for permission on every command. How do I reduce prompts?

Use claude config to adjust permission settings. You can allow specific tools (like Bash, Read, Write) to run without prompting. The --dangerously-skip-permissions flag exists but is not recommended for general use.

What is the difference between CLAUDE.md and .claude/settings.json?

CLAUDE.md provides natural-language context and instructions (architecture rules, coding standards). .claude/settings.json is for machine configuration: MCP servers, permission rules, and environment variables.

Can I use Claude Code with VS Code or JetBrains?

Yes. Claude Code has official extensions for both VS Code and JetBrains IDEs. Install the extension from the marketplace, and it connects to the same CLI under the hood. You can also use it as a standalone desktop app or via the web at claude.ai/code.

Does Claude Code work on Windows without WSL?

No. Claude Code requires a Unix-like environment. On Windows, you must use WSL2. For best performance, keep your project files inside the WSL filesystem (~/) rather than on mounted Windows drives (/mnt/c/).

How do I keep costs down?

Use /compact frequently to compress conversation history. Use /clear between unrelated tasks. Break large tasks into focused steps instead of one massive prompt. Toggle /fast mode for routine tasks that don't need extended thinking.
